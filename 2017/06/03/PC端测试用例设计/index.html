PC 端测试用例设计

非业务测试用例设计

【稳定性测试】
程序长时间运行情况下
网络不稳定情况下
电压不稳定情况下
系统休眠情况下
磁盘不稳定，如 U盘/移硬被意外插拔等情况
被其他程序干扰，如病毒木马环境

【健壮性测试】
注册表相关键值操作时不允许写入或读取的情况
Xml 组件被反注销的情况
Com 组件初始化失败的情况
Clsid 注册不成功的情况
文件相互覆盖导致模块不匹配的情况
核心模块丢失的情况
规则或数据文件被破坏的情况
文件内容超大的情况
磁盘空间不足的情况
磁盘目录被 ntfs 加密的情况

【安全性测试】
和服务端交互接口安全性
程序调用安全性
Ring3 程序安全性－fuzz 工具攻击
网络通讯加密
数据文件加密
服务端数据存储加密

【安装、卸载测试】
单独安装测试
覆盖安装低版本、相同版本、低版本覆盖高版本
单独安装在默认目录
单独安装在非默认目录
管理员权限下安装
非管理员权限下安装
超常目录
特殊字符等目录
控制面板、开始菜单、程序目录下等常见入口卸载测试
卸载后生成 pending 文件后，不重启再安装的情况

【兼容性测试】
同类软件兼容性测试
不同操作系统平台兼容
不同的浏览器兼容
复杂网络环境兼容性
程序的高低版本自身的兼容性

【升级测试】
数据文件升级
程序文件升级
大的升级包升级
外网低版本向此版本升级
本版本向更高版本升级
不同的升级方式

【UI 及用户体验测试】
中文简体、繁体、英文环境下的界面展现
宽屏、上网本、pad 的 ui 界面展现
大字体下的 ui 展现
不同分辨率下的展现
各界面元素文字、图片显示正确
各界面元素状态正确
界面美观
布局合理
执行有风险操作时，有确认或放弃等提示语
图标显示美观，放大镜放大少许后无毛刺、锯齿
字体颜色和大小美观协调
窗口切换、移动、改变大小时显示正常
键盘操作支持
操作简单、使用方便，功能操作顺序合理

测试用例设计策略

【设计思想】
需求覆盖度
代码覆盖度
深挖本质
一点多例

【需求覆盖度】
前提熟读和理解需求，了解产品背景,完成需求测试
需求分解，一级需求、二级需求、三级需求，不同级别需求测试力度
不同，用例设计力度不同
根据具体级别需求再细分子需求，直到不能细分为止
测试用例设计时先搭建用例框架，从大的需求点入手，逐步将用例
覆盖到具体子需求
每一个需求都是可测试，有具体的case支持

【代码覆盖度】
根据需求，了解产品代码逻辑,掌握具体实现方法，包括调用细节
有哪些参数
开发人员的代码 review，确保没有增加额外代码
开发的白盒测试以及单元测试
测试用例设计时先搭建用例框架，从核心功能逻辑点入手，逐步将用例覆盖到具体子需求，细微逻辑处理
每一个逻辑点都是可测试，有具体的case支持

【自动化测试支持】

【深挖本质】
每一个用例设计前都要想想这个 case 的本质是什么，是要验证什么问题，里面的代码可能是怎么写的
case 中每一步的操作是否合理，从执行效率上来讲
多个 case 之间需要考虑执行优先级
多个 case 时，需要考虑那些 case 可能是冗余的

【一点多例】
每一个点都肯定不是一条 case 能验证，就像任何事务必然有多个属性
设计时需要根据属性不同设计多个 case，正面、反面，正常、异常
性能、效率
